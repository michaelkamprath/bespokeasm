# Operand Labels Feature Project Plan

## 1. Objective
Add **operand labels** to BespokeASM so source code can define labels on emitted operand argument bytes directly in instruction syntax, enabling clean self-modifying code workflows without separate data directives.

Primary user-driven example:

```asm
normal_label:
    STB @operand_label: 0xFFFF
```

The label `operand_label` should resolve to the bytecode address of the encoded operand argument value (here, the first byte of `0xFFFF`).

---

## 2. Scope

### In Scope
- New assembly syntax for operand labels.
- Core assembler parsing + symbol definition behavior.
- Support for these operand types:
  - `numeric`
  - `indirect_numeric`
  - `deferred_numeric`
  - `address`
  - `relative_address`
- Multi-operand support (multiple operand labels in one instruction).
- Clear diagnostics for invalid uses.
- Documentation updates.
- Language extension behavior updates generated by `generate-extension` (hover + syntax/semantic highlighting where relevant).
- Operand-label syntax highlighting split into independently styleable elements for `@`, operand-label name, and `:`.
- Unit/integration tests.

### Out of Scope (Initial Release)
- Operand labels on operand types not listed above.
- Operand labels in instruction macro definitions.
- Operand labels in instruction macro invocations.
- New ISA configuration schema changes (feature should be syntax/runtime behavior only).
- Binary format changes.

## 3. Proposed Syntax and Semantics (Implementation Contract)

### 3.1 Syntax

Operand label marker is `@name:` and must be attached to the **argument expression**.

```asm
; numeric / address / relative_address (non-curly)
MNEMONIC @label: expr

; indirect_numeric
MNEMONIC [@label: expr]

; deferred_numeric
MNEMONIC [[@label: expr]]

; relative_address (curly form, if configured)
MNEMONIC {@label: expr}
```

Valid spacing examples:
- `@lbl:expr`
- `@lbl: expr`

Invalid spacing/syntax examples:
- `@lbl : expr` (whitespace before `:` is not allowed)
- `@lbl expr` (missing `:`)

### 3.2 Semantics
- Operand label points to the first emitted byte/word position of that operand's encoded argument bytes.
- For `indirect_numeric` and `deferred_numeric`, label points to encoded pointer value, not runtime dereferenced target.
- For `relative_address`, label points to emitted encoded relative offset bytes, not absolute target address.
- Operand value is still required (operand labels are annotation, not value inference).

### 3.3 Multi-Operand Behavior
- One operand label max per operand.
- Multiple operand labels per instruction are allowed.
- Label addresses are based on **final emitted argument order** (including `reverse_argument_order` behavior).

### 3.4 Invalid Uses
- Using operand labels on unsupported operand types should produce specific error.
- Label on operand that emits no argument bytes should error.
- Label on operand whose emitted argument bytes are not word-aligned or not word-full should error.
- Multiple operand-label annotations on the same operand should error (even if names differ).
- Operand labels are invalid in instruction macro definitions and in instruction macro invocations.
- Duplicate label definition should follow existing duplicate-label error behavior.
- Operand-label annotation and operand expression must appear on the same physical source line.
- Missing `:` after operand-label name (for example `@lbl expr`) should produce a malformed operand-label syntax diagnostic.

---

## 4. Resolved Design Decisions

1. **Local-scope anchor semantics**
   - Operand-label definitions follow existing BespokeASM label scope rules.
   - Scope is determined by the same established prefix conventions used by normal labels (`_`, `.`, and named-scope prefixes where configured).
   - No special-case scoping behavior is introduced for operand labels.

2. **Non-word-aligned argument starts**
   - Operand labels are only valid when the annotated operand argument emits a word-addressable value.
   - If the argument starts at a non-word-aligned position, or if the argument emission is not word-full, compilation must fail with a targeted error.

3. **Colon spacing tolerance**
   - Accept both `@lbl:expr` and `@lbl: expr`.
   - Do not allow whitespace between operand-label name and colon (for example, `@lbl : expr` is invalid).
   - Operand label and operand expression must remain on the same physical line.
   - Missing `:` after operand-label name (for example, `@lbl expr`) is invalid and should produce targeted malformed-syntax guidance.

4. **Multiple operand labels on one operand**
   - Not allowed.
   - Exactly one operand-label annotation may appear per operand.
   - Rationale: aligns with simpler syntax and avoids extra parser/bytecode-mapping complexity.

## Phase B - Core Assembler Feature

### B1. Operand-label parsing helper
Create a shared parsing helper (new module recommended), e.g.:
- `src/bespokeasm/assembler/model/operand/operand_label.py`

Responsibilities:
- Detect optional `@label:` annotation from raw operand string.
- Validate label name against existing label rules (`is_valid_label`, register-name exclusion).
- Return structured result:
  - cleaned operand string (annotation removed)
  - optional label name
  - annotation metadata (for diagnostics)
- Support wrapper-specific forms (`@lbl: expr`, `[@lbl: expr]`, `[[@lbl: expr]]`, `{@lbl: expr}`).

### B2. Supported operand type integration
Integrate helper into parsing of:
- `numeric` (via `numeric_expression.py`)
- `address` (inherits numeric expression path)
- `indirect_numeric` (`indirect_numeric.py`)
- `deferred_numeric` (`deferred_numeric.py` via indirect behavior)
- `relative_address` (`relative_address.py`)

Implementation notes:
- Ensure existing matching/validation still applies after annotation removal.
- Preserve existing expression semantics.
- Ensure annotation is syntactically required to be inside wrappers for wrapped operand types.

### B3. Carry label metadata through parse/generation pipeline
Likely changes:
- Extend `ParsedOperand` in `src/bespokeasm/assembler/model/operand/__init__.py` with optional operand-label field.
- Ensure matched operand data preserves label->argument-part association.
- Ensure unsupported operand types with annotation fail deterministically with explicit diagnostic.

### B4. Compute label addresses from emitted parts
Key files:
- `src/bespokeasm/assembler/bytecode/generator/instruction.py`
- `src/bespokeasm/assembler/line_object/instruction_line.py`
- `src/bespokeasm/assembler/engine.py`

Approach:
- Maintain mapping from operand label names to concrete argument `ByteCodePart` instances.
- During pass-1 (when line start address is known), compute each operand label address by walking final emitted parts and alignment rules.
- Register operand labels in scope using same named-scope/global/file/local machinery used for address labels (without changing constant behavior).

### B5. Diagnostics
Add targeted errors (avoid generic "no valid operands configured" where possible):
- Unsupported operand type for operand-label annotation.
- Annotated operand has no argument bytes.
- Malformed operand-label syntax.
- Missing `:` after operand-label name (for example `@lbl expr`).
- Multiple operand-label annotations on one operand (regardless of label name).
- Non-word-aligned or non-word-full operand emission for operand label target.
- Operand label annotation split across lines from its operand expression.

### B6. Core tests (implemented in this phase)
Add/update tests and fixtures:
- `test/config_files/test_operand_labels.yaml` (new)
- `test/test_instruction_parsing.py`
- `test/test_line_objects.py` (if needed for parse behavior)
- `test/test_engine.py` (if needed for pass-1 label address correctness)

Required coverage:
1. `numeric` operand label basic case.
2. `address` operand label basic case.
3. `relative_address` label case (curly and non-curly if config supports both).
4. `indirect_numeric` and `deferred_numeric` wrapper placement cases.
5. Multi-operand instruction with two operand labels.
6. `reverse_argument_order` mapping correctness.
7. Multiple operand-label annotations on the same operand errors.
8. Unsupported operand type errors.
9. Missing/invalid annotation syntax errors.
   - Include missing-colon case (`@lbl expr`) and whitespace-before-colon case (`@lbl : expr`).
10. Duplicate label name errors.
11. Scope behavior (global/file/local or chosen constraints).
12. Instruction macro definitions and macro invocations reject operand-label syntax.

### Phase B Exit Criteria
- Valid operand-label syntax parses for all supported operand types.
- Invalid forms in Section 3.4 fail with targeted diagnostics (not only generic operand matching failures).
- Operand-label addresses resolve correctly in both normal and `reverse_argument_order` layouts.

## 9. Risks and Mitigations

- **Risk:** Ambiguous scope behavior for operand labels.
  - **Mitigation:** Lock scope behavior in docs before coding; add explicit tests.

- **Risk:** Generic operand matching hides useful errors.
  - **Mitigation:** Capture and prioritize operand-label-specific diagnostics.

- **Risk:** Extension regexes become brittle with complex inline syntax.
  - **Mitigation:** Move from single-match line-start patterns to explicit multi-definition scanning + tests.

- **Risk:** Emitted-part address computation mismatches final byte stream.
  - **Mitigation:** Reuse same alignment/order model used for instruction word packing and test against reverse-order cases.
